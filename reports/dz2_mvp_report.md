# Отчёт по ДЗ 2 (MVP проекта)

## 1) Что сделано на этапе MVP

Реализован рабочий MVP сервиса "Въездник": пользователь выбирает страну, задаёт вопрос в чате и получает структурированный ответ по правилам въезда и документам.

Состав MVP:

- фронтенд на Next.js с выбором страны и чат-интерфейсом;
- backend на FastAPI с endpoint `POST /chat`;
- мультиагентная логика ответа (`guard`, `router`, `lawyer`, `form`) под управлением LangGraph;
- RAG-поиск по базе знаний в Qdrant;
- JSON-метаданные с формами/ссылками;
- скрипты `parsing.py` и `ingest.py` для подготовки и индексации данных;
- базовые логи запросов и ошибок (`chat.log`, `errors.log`) с маскированием ПДн.

## 2) Почему выбран такой стек

- **FastAPI**: быстрый API-слой и удобная валидация моделей.
- **Next.js + TypeScript**: быстрое создание UI и хорошая типизация клиентской части.
- **LangGraph + LangChain**: разделение ответа на специализированные роли агентов и управляемая маршрутизация.
- **Qdrant**: простой и быстрый векторный поиск для MVP на небольшом количестве стран.
- **Docker Compose**: воспроизводимый локальный запуск одной командой.

## 3) Ключевые решения

- Отделили фактические правила въезда (`lawyer_agent`) от работы с формами (`form_agent`), чтобы уменьшить смешивание логики.
- Добавили входной guard для off-topic запросов.
- Температура генерации выставлена в 0 для более детерминированных ответов.
- Источники данных разделены на markdown (факты) и JSON (точные ссылки/формы).

## 4) Сложности и как решали

1. **Официальные сайты блокируют парсинг** (403, anti-bot/WAF):
   - добавлен каскадный fetch (`requests` -> `undetected-chromedriver` -> `selenium fallback`);
   - добавлен детектор interstitial-страниц антибота, чтобы не сохранять мусор в базу знаний.

2. **Долгий старт backend из-за тяжёлых импортов**:
   - перенесён импорт оркестратора в обработчик чата (lazy import), чтобы `/health` поднимался сразу.

3. **Пограничные случаи в роутинге**:
   - введена нормализация неизвестного intent в `BOTH`.

## 5) Что осталось на будущее

- расширить набор стран и покрытие источников;
- добавить более формализованную offline-оценку качества RAG (gold set);
- добавить UI-метрики качества ответа (лайк/дизлайк).

## 6) Оценка времени и стоимости с LLM

- **Время на этап MVP**: около **32-38 часов** суммарно (команда).
- **Использование генеративных моделей/ассистентов**:
  - проектирование промптов и структуры агентов;
  - ускорение написания шаблонного кода и рефакторинга.
- **Оценка стоимости этапа**:
  - подписка на код-ассистент: ориентировочно **$20-25/мес** на участника;
  - LLM API (Mistral/OpenAI-совместимый): ориентировочно **$8-20** за этап MVP при умеренном объёме тестов;
  - суммарно по этапу: примерно **$30-45** (оценка, зависит от интенсивности тестирования).
