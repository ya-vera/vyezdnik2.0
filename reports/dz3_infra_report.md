# Отчёт по ДЗ 3 (Инфраструктура проекта)

## 1) Что сделано на инфраструктурном этапе

На этапе ДЗ3 проект переведён в состояние, удобное для локального деплоя и наблюдаемости:

- собран единый `docker-compose` стек для сервисов приложения;
- добавлены инфраструктурные сервисы: Redis, Qdrant, Prometheus, Grafana, Blackbox Exporter;
- настроен мониторинг backend, Qdrant и synthetic health-check;
- добавлены кастомные метрики мультиагентного оркестратора;
- настроено автоподключение datasource и dashboard в Grafana через provisioning;
- включено хранение сессионной истории в Redis с TTL (30 минут простоя).

## 2) Почему выбраны такие решения

- **Docker Compose**: позволяет поднимать весь стек одной командой, без ручной настройки зависимостей.
- **Redis**: лёгкое in-memory хранилище для сессий с TTL, соответствует требованиям stateless API + временная память диалога.
- **Prometheus + Grafana**: стандартная, простая и понятная связка для метрик и визуализации.
- **Blackbox Exporter**: проверяет доступность API как внешний клиент (не только внутренние метрики процесса).
- **Qdrant в отдельном контейнере**: изоляция векторной БД и предсказуемый lifecycle данных.

## 3) Что именно мониторится

- доступность backend (`up`) и synthetic probe `/health`;
- HTTP-метрики FastAPI (`/metrics`);
- кастомные метрики агентов:
  - `vyezdnik_agent_calls_total`;
  - `vyezdnik_agent_duration_seconds`;
  - `vyezdnik_orchestrator_runs_total`;
- метрики Qdrant (`/metrics`);
- recording rules для стабильных графиков в Grafana.

## 4) Сложности и как решали

1. **Конфликты Docker контейнеров по именам**
   - повторные запуски давали `Conflict. The container name is already in use`;
   - решено через корректный lifecycle (`docker compose down/up`) и очистку конфликтующих контейнеров.

2. **Долгая инициализация backend при старте**
   - тяжёлые импорты и модели тормозили доступность endpoint;
   - применён lazy import оркестратора в `POST /chat`, чтобы health-check работал сразу.

3. **Некачественные данные парсинга из anti-bot страниц**
   - добавлен детектор interstitial/WAF-страниц, чтобы не индексировать мусор в knowledge base.

## 5) Что можно усилить дальше

- добавить Alertmanager и базовые alert-правила;
- добавить централизованные логи (например, Loki);
- добавить tracing (OpenTelemetry) для end-to-end запросов;
- вынести секреты в более безопасный способ хранения для production.

## 6) Оценка времени и стоимости с LLM

- **Время на этап ДЗ3**: около **20-26 часов** суммарно.
- **Использование генеративных моделей/ассистентов**:
  - ускорение настройки Docker/monitoring-конфигов;
  - помощь в рефакторинге и проверке интеграции сервисов.
- **Оценка стоимости этапа**:
  - подписка на код-ассистент: ориентировочно **$20-25/мес** на участника;
  - LLM API на отладку интеграции: ориентировочно **$5-12**;
  - суммарно по этапу: примерно **$25-37** (оценка).
